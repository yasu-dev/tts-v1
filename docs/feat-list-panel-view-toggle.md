# feat: 一覧表示のリスト/パネル切替機能

## 1. 修正内容の詳細化

### 概要

指揮本部（`/command`）、DMAT（`/transport`）、搬送部隊（`/transport-team`）の3つのダッシュボードに、患者一覧の表示形式をリスト形式とパネル形式で切り替える機能を追加する。

### 目的

- **リスト形式（現行）**: コンパクトな1行表示。大量の患者を素早くスキャンするのに適する
- **パネル形式（新規）**: カード型の拡張表示。2つの本質的価値がある:
  1. リスト形式では見えないバイタルサイン・症状・主訴を一覧上で確認できる
  2. 複数患者を同時に視野に入れ、横断比較で優先度・状況を判断できる

### Before / After

**Before（リスト形式のみ）**:

```
[T-001 赤] 📍現場 | 45歳 男性
            発見: 山田太郎 | 新宿区西新宿6丁目
                                        [詳細] [搬送割当]
```

- 表示情報: タグ番号、トリアージ区分、フェーズ、年齢、性別、担当者、位置

**After（パネル形式を追加）**:

パネル形式はロール別に表示内容を最適化する（詳細は後述）。各ロールの意思決定に最も重要な情報を優先的に配置し、3層スキャンモデル（Glance → Scan → Read）で情報階層を構成する。

### 切替UI

患者一覧セクション（`<h2>患者一覧</h2>`）の右側にアイコン2つを横並びで配置:

```
患者一覧（12件）                    [≡] [⊞]
```

- `≡`（リストアイコン）: リスト形式に切替（アクティブ時に背景色付き）
- `⊞`（グリッドアイコン）: パネル形式に切替（アクティブ時に背景色付き）
- アイコンはインラインSVGで実装（新規依存なし）
- 選択状態は `localStorage` に保存し、ページリロード後も復元

### レスポンシブグリッド配置

現行仕様（PC推奨、タブレット・スマホ対応）に準拠し、画面幅で列数を切替:

| 画面幅                       | 列数 | Tailwind         |
| ---------------------------- | ---- | ---------------- |
| ~767px（スマホ）             | 1列  | `grid-cols-1`    |
| 768~1023px（タブレット縦等） | 2列  | `md:grid-cols-2` |
| 1024px~（PC、タブレット横）  | 3列  | `lg:grid-cols-3` |

カード高さは内容に応じた自然な高さ（固定しない）。

### ロール別パネル設計

#### 設計原則：3層スキャンモデル

パネルカードは人間の視線動作に合わせた3層構造で情報を配置する:

| 層                   | 所要時間 | 目的                   | 表現手段                           |
| -------------------- | -------- | ---------------------- | ---------------------------------- |
| **Glance**（一瞬）   | ~0.5秒   | 重症度と状態を色で判別 | 左ボーダー色 + 背景薄色 + タグ番号 |
| **Scan**（ざっと）   | ~2秒     | 比較対象の数値を読む   | 太字の数値、異常値の赤字           |
| **Read**（じっくり） | ~5秒     | 詳細文脈を把握         | 通常サイズのテキスト               |

#### カード外観（共通）

左ボーダー(4px) + 背景薄色 でトリアージ区分を視覚的に即時判別:

| 区分 | ボーダー                       | 背景              |
| ---- | ------------------------------ | ----------------- |
| 赤   | `border-l-4 border-red-500`    | `bg-red-50/60`    |
| 黄   | `border-l-4 border-yellow-400` | `bg-yellow-50/60` |
| 緑   | `border-l-4 border-green-500`  | `bg-green-50/60`  |
| 黒   | `border-l-4 border-gray-900`   | `bg-gray-100`     |

#### バイタル異常値アラート（共通）

異常値は赤字で表示し、一覧上で即座に危険な患者を識別可能にする:

| 項目         | 正常範囲  | 異常（赤字表示）  |
| ------------ | --------- | ----------------- |
| SpO2         | 95%以上   | 94%以下           |
| 血圧(収縮期) | 90以上    | 89以下            |
| 脈拍         | 50-120    | 49以下 or 121以上 |
| 呼吸数       | 10-29     | 9以下 or 30以上   |
| 意識         | A（清明） | V=黄字、P/U=赤字  |

#### 経過時間の表示仕様（共通）

**表示フォーマット**: `経過 {N}分`

- 「経過」プレフィックスを常に付与し、この数字が何を意味するか即座に認知可能にする
- 計算式: `now - created_at`（タグ登録からの経過時間）
- 更新頻度: 1分ごとに再計算

**色分け:**

| 経過時間 | 色                | 意味 |
| -------- | ----------------- | ---- |
| 15分未満 | `text-gray-500`   | 通常 |
| 15-30分  | `text-yellow-600` | 注意 |
| 30分以上 | `text-red-600`    | 警告 |

#### 症状・傷病名の表示マッピング（共通）

`conditions[]` の英語enum → 日本語表示:

| enum値       | 表示                                   |
| ------------ | -------------------------------------- |
| `contusion`  | 打撲                                   |
| `fracture`   | 骨折                                   |
| `sprain`     | 捻挫                                   |
| `amputation` | 切断                                   |
| `burn`       | 熱傷                                   |
| `other`      | その他（`condition_other` の値を表示） |

複数ある場合はカンマ区切り（例: 「骨折, 熱傷」）。未入力時は「-」。

#### 意識レベル（AVPU）の表示マッピング（共通）

`vital_signs.consciousness.avpu` の値 → 表示:

| enum値         | 表示 | 文字色                  |
| -------------- | ---- | ----------------------- |
| `alert`        | A    | 通常（`text-gray-900`） |
| `voice`        | V    | 黄（`text-yellow-600`） |
| `pain`         | P    | 赤（`text-red-600`）    |
| `unresponsive` | U    | 赤（`text-red-600`）    |

### 指揮本部パネル

**意思決定文脈**: 「この現場全体で、今どの患者が最もリスクが高く、搬送の優先度が高いか？」

**比較軸**: 重症度 × バイタルの深刻さ × 滞留時間

```
┌─────────────────────────────────────┐
│▌ T-001  📍現場          経過 32分   │ Glance: タグ番号, フェーズ, 経過時間
│▌ 45歳 男性                         │
│▌───────────────────────────────────│
│▌ SpO₂ 94%⚠   脈拍 110             │ Scan: バイタル4項目
│▌ 血圧 90/60⚠  意識 V              │   異常値は赤字で即座に視認
│▌───────────────────────────────────│
│▌ 骨折, 熱傷      発見: 山田太郎    │ Read: 症状 + 担当者
│▌                  [詳細] [搬送割当] │ Action
└─────────────────────────────────────┘
```

| ゾーン | 表示項目                                      | 理由                                  |
| ------ | --------------------------------------------- | ------------------------------------- |
| Glance | タグ番号、フェーズ(icon+名)、経過時間(色分け) | 滞留時間が長い患者を即座に検知        |
| Scan   | SpO2、脈拍、血圧、意識レベル（4項目）         | バイタルの比較で搬送優先度を判断      |
| Read   | 症状（短縮表示）、担当者                      | 状況の補足（1行に並べてコンパクト化） |
| Action | [詳細] [搬送割当]（未割当時のみ）             | 既存と同じ                            |

**設計判断**:

- 呼吸数は省略（SpO2で代替可能、表示面積節約）
- 主訴は省略（指揮本部は個別の臨床詳細より全体俯瞰が優先）
- 経過時間 = `now - created_at`（登録からの経過）。表示は「経過 {N}分」フォーマット

### DMATパネル

**意思決定文脈**: 「応急救護所にいるこの患者たちを、それぞれどの病院のどの診療科に送るべきか？」

**比較軸**: 傷病種別（搬送先選択）× バイタル重症度（搬送優先度）

```
┌─────────────────────────────────────┐
│▌ T-001  🏥応急           経過 15分   │ Glance: タグ番号, フェーズ, 経過時間
│▌ 45歳 男性                         │
│▌───────────────────────────────────│
│▌ 呼吸 28⚠  SpO₂ 94%⚠  意識 V    │ Scan: バイタル5項目（医療者向け）
│▌ 脈拍 110   血圧 90/60⚠          │
│▌───────────────────────────────────│
│▌ 骨折, 熱傷                        │ Read: 症状（搬送先選択の決定的情報）
│▌ 主訴: 右大腿部痛                  │ Read: 主訴
│▌                    [詳細] [選択]  │ Action
└─────────────────────────────────────┘
```

| ゾーン | 表示項目                                      | 理由                                           |
| ------ | --------------------------------------------- | ---------------------------------------------- |
| Glance | タグ番号、フェーズ(icon+名)、経過時間(色分け) | 経過時間の長い患者を検知                       |
| Scan   | 呼吸数、SpO2、意識、脈拍、血圧（5項目）       | DMATは医療チーム。呼吸数は搬送リスク判断に必須 |
| Read   | 症状・傷病名 + 主訴                           | 「この患者はどの診療科に送るか？」の判断に直結 |
| Action | [詳細] [選択]                                 | 既存と同じ                                     |

**設計判断**:

- 呼吸数を追加（DMATは医療者、呼吸数は搬送リスクの主要指標）
- 症状と主訴を独立行で表示（搬送先選択の最重要情報）
- 経過時間 = `now - created_at`（登録からの経過）。表示は「経過 {N}分」フォーマット

### 搬送部隊パネル

**意思決定文脈**: 「自チームが担当する患者の搬送フローは今どうなっていて、次に何をすべきか？」

**比較軸**: 搬送ステータス × 担当チーム × 重症度

```
┌─────────────────────────────────────┐
│▌ T-001  ⏳割当済                    │ Glance: タグ番号 + ステータス
│▌ 45歳 男性         新宿ポンプ1     │ 患者識別 + チーム名
│▌───────────────────────────────────│
│▌ SpO₂ 94%⚠  意識 V                │ Scan: バイタル最小限（2項目）
│▌───────────────────────────────────│
│▌ 骨折, 熱傷                        │ Read: 症状
│▌ 📍 新宿区西新宿6丁目             │ Read: 搬送元の位置
│▌                   [詳細] [搬送開始]│ Action
└─────────────────────────────────────┘
```

| ゾーン | 表示項目                                     | 理由                               |
| ------ | -------------------------------------------- | ---------------------------------- |
| Glance | タグ番号、ステータス(割当済/搬送中)          | 搬送フロー進捗が最重要             |
| (補助) | チーム名（右寄せ）                           | 「自分のチームの患者」を即座に識別 |
| Scan   | SpO2、意識レベル（2項目のみ）                | 搬送中の急変リスク判断に最低限必要 |
| Read   | 症状 + 位置情報                              | 搬送元の把握                       |
| Action | [詳細] [搬送開始]/[OK/キャンセル]/[応急到着] | 既存と同じ                         |

**設計判断**:

- バイタルは2項目のみ（搬送部隊は消防隊員が多く、呼吸数・脈拍・血圧の詳細は行動判断に不要）
- チーム名を上部に配置（自チームのフィルタリングはドロップダウンで行うが、パネル上でも確認可能）
- 位置情報を表示（搬送元=迎えに行く場所の把握）
- 経過時間は省略（搬送部隊はフェーズ進行が最重要で、滞留時間の比較は二次的）

## 3. 影響範囲の分析

### 既存機能への影響

| 機能                                     | 影響     | 対応                                                                                                                                                   |
| ---------------------------------------- | -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| フィルタリング（トリアージカテゴリ）     | 影響あり | `filteredTags` をそのまま使用するため、リスト/パネル両方でフィルタが正しく動作する。変更不要                                                           |
| ステータスフィルター（指揮本部）         | 影響あり | 同上                                                                                                                                                   |
| 搬送部隊フィルター（搬送部隊）           | 影響あり | 同上                                                                                                                                                   |
| 地図連動（指揮本部）                     | 影響あり | `selectedPatientId` のハイライト表示はリスト/パネル両方で `id` ベースで対応。PatientListItemとPatientPanelCardの両方に `id={patient-${tag.id}}` を付与 |
| 詳細モーダル                             | 影響なし | 既存の `PatientDetailModal` を引き続き使用。`onDetailClick` 経由で呼び出し                                                                             |
| アクションボタン（搬送割当、搬送開始等） | 影響なし | `actions` prop で各ダッシュボードから渡すため、ロジック変更なし                                                                                        |
| リアルタイム更新                         | 影響なし | tags state の更新処理はダッシュボード側で変更なし。コンポーネントは props で受け取るだけ                                                               |
| ページネーション（DMAT）                 | 影響なし | `currentPageTags` をそのまま使用                                                                                                                       |
| QRスキャン                               | 影響なし | 切替UIとは独立した機能                                                                                                                                 |

### API・DB・外部連携への影響

なし。本変更はフロントエンドのUI表示のみで、データ取得ロジック・API呼び出し・DBスキーマに変更はない。

### 非機能への影響

| 観点             | 影響                                               | 評価                                                                                             |
| ---------------- | -------------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| パフォーマンス   | パネル形式は1患者あたりのDOM要素が増加             | 現行の患者数規模（数十〜数百件）では問題なし。パネルカード内は条件分岐でnullを返すだけなので軽量 |
| メモリ           | viewMode state + localStorage 読み書き追加         | 無視できるレベル                                                                                 |
| レスポンシブ     | パネル形式でグリッドレイアウト追加                 | Tailwind CSS の標準グリッドで対応。追加のメディアクエリ不要                                      |
| アクセシビリティ | 切替ボタンに `aria-label` と `aria-pressed` を付与 | スクリーンリーダー対応                                                                           |
| セキュリティ     | 影響なし                                           | 表示のみの変更、入力やデータ送信なし                                                             |
