一覧画面の設計ポイント

  • サムネイル重視: 現場図の主な識別手段はサムネイルの内容。Konva stage.toDataURL() でキャプチャし、カードの75%を占めるサイズで表示。名前を読まなくても「どの図か」を瞬時に視覚判断できる
  • カード情報: デフォルト名 + 最終更新時刻のみ。作成者名は表示しない（隊長クラスに固定のため不要）
  • デフォルト名: 現場図 MM/DD HH:MM（作成日時ベース）。現場では名前を変更する余裕がないことを前提とし、変更されなくても時系列で識別可能な命名
  • 名前変更: 一覧画面でカード名をタップ → インライン編集。キャンバス画面では名前変更不可（描画に集中させる）
  • レイアウト: タブレット縦持ちで2列×2行。想定2〜3ファイル + 新規カード = 最大4枚が一画面に収まる。仮に超えても縦スクロールで十分
  • 新規作成: 常に先頭に＋カードを表示。1タップで空キャンバスへ遷移
  • ソート: 更新日時降順（最近使った図が先頭、新規カードの次）
  • 閲覧ロール（TRI/TRN等）: 同じ一覧画面だが「新規作成」カード非表示。既存図の閲覧のみ
  • 削除: カード長押し → 確認ダイアログ（誤削除防止）

サムネイルサイズの設計根拠

サイズ決定: タブレット縦持ち（768×1024px）を基準。
・ヘッダー: 56px / 上下余白: 32px / カード間隔: 16px → 利用可能高さ: 920px
・2列: (768 - 32 - 16) / 2 = 360px幅 / カード
・アスペクト比 4:3 → サムネイル高さ: 270px
・名前+更新時刻: 52px → カード総高さ: 322px
・2行: 322×2 + 16 = 660px → 920pxの72%で余裕を持って収まる
・サムネイルはカード面積の約84%を占め、内容を視覚的に判断するのに十分な大きさ

③ 自動保存の導入

災害現場で「保存し忘れ」は致命的。操作に集中するために自動保存へ

自動保存の設計

BEFORE
  • 手動保存（ヘッダーの「保存」ボタン）
  • 未保存状態のisDirtyフラグ管理
  • 閉じる時に「未保存の変更があります」確認ダイアログ
  • 問題: 災害現場で保存忘れ → データ消失リスク
AFTER
  • デバウンス自動保存（変更後2秒で自動保存）
  • 保存ボタン廃止 → キャンバス領域拡大
  • 確認ダイアログ不要 → 閉じるが即座に完了
  • 自動保存インジケーター（✓ 保存済み）表示

自動保存のタイミング

トリガー保存タイミング理由
アイコン配置・移動・削除操作後 2秒連続操作を妨げない
テキスト編集入力停止後 2秒タイピング中に保存しない
回転・Z順序変更操作後 2秒デバウンスで集約
キャンバスから離脱即座に保存「戻る」「閉じる」時に確実に保存

自動保存インジケーター

● 保存済み
● 保存中...
● 保存失敗（リトライ中）
実装: useEffect + setTimeout（2000ms デバウンス）。data の変更を検知して自動でSupabase UPSERTを実行。保存エラー時は3回リトライ後にトースト通知。

④ コンテキストメニュー化

下部ツールバー廃止 → アイコン直接操作でキャンバス領域を最大化

操作方式の比較

BEFORE — 下部ツールバー方式
  • アイコン選択 → 視線を下部ツールバーに移動 → 操作ボタン押下
  • 常に画面下部を占有（60px）
  • 選択状態がツールバーに表示（アイコンから目が離れる）
  • 問題: 視線移動が多く、災害現場での操作に不向き
AFTER — コンテキストメニュー方式
  • アイコンを長押し（500ms） → アイコン直上にメニュー表示
  • 画面下部を完全開放 → キャンバス最大化
  • 視線がアイコンから離れない（操作と対象が同一位置）
  • Excalidraw / tldraw と同等の操作性

オフライン耐性

災害現場では通信が不安定・断絶する可能性が高い。自動保存がネットワークエラーで失敗するケースへの対策が必須。
状況対策
保存失敗ローカル（localStorage）にバッファリング → 通信復旧時に自動リトライ（最大3回、指数バックオフ）
完全オフラインlocalStorageに最新データを常時保持。オンライン復帰時にサーバーと同期
保存失敗通知赤インジケーター「● 保存失敗」+ 「ローカルに保存済み」サブテキスト
競合（同時編集）MVPでは後勝ち（Last Write Wins）。将来的にリアルタイム同期検討
